# GitHub Copilot Instructions

This repository enforces a plan-first, spec-driven execution model. Follow this workflow for all work:

## Default Execution Sequence

1. **Start from context**: Plan, existing task spec, or clearly scoped task request
2. **Derive specs first**: If a plan or phase slice exists, create or refine task specs before implementing
   - Use `ai/skill/plan-to-spec.md` as the default workflow for spec generation
3. **Implement only after clarity**: Do not edit until the task boundary is clear
4. **Validate explicitly**: Confirm the work meets the spec before completion
5. **Write back stable facts**: Only write to `ai/doc/facts/*` when information is stable and reusable
6. **Promote skills when ready**: Move repeated workflows to `ai/skill/*` when they stabilize

## Key Rules

- **Spec-first is mandatory**: Only skip spec creation for trivially narrow, already spec-complete requests
- **One spec, one outcome**: Prefer a single task spec with one primary reviewable outcome and a short checklist
- **Keep specs narrow**: Refine existing specs when iterating within the same reviewable slice
- **Guard boundaries**: Do not let task execution expand project or phase scope without explicit replan
- **No speculative abstractions**: Implement only what the spec requires—nothing more
- **Documentation is targeted**: Plans may remain temporary; the task spec is the durable execution artifact

## When the Task is Unclear

- If no plan exists: Ask for plan context OR generate a draft task spec in `ai/doc/specs/` for your review
- If the boundary shifts: Create a new dated spec first before changing course
- If spanning multiple phases: Keep the hierarchy explicit as `project_target -> current_target -> phase -> plan -> task`

## Workspace Structure Reference

- `AGENTS.md` – Repository-level operating rules (canonical source)
- `ai/doc/specs/` – Task specifications (where implementation specs live)
- `ai/doc/facts/` – Stable, reusable project context
- `ai/skill/` – Reusable workflows and shared patterns
- `.cursor/rules/` – Cursor-specific enforcement rules (mirrors this workflow)
- `project/` – Human-facing project control surface

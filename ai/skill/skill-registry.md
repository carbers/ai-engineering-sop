# Skill Registry

Keep this registry in sync with the files in `ai/skill/`.
When adding, removing, or renaming a skill file, update this registry in the same change.

| Name | Purpose | Trigger | Inputs | Outputs | Status |
|---|---|---|---|---|---|
| `plan-to-spec` | Convert a plan or phase slice into one or more narrow task specs | A plan or phase slice exists and work is about to move into implementation | plan or phase slice, current context, constraints, risks, and for phase-aware work the current target and plan level | one or more specs with scope, status, checklist, done condition, validation, write-back guidance, and when needed parent mapping plus failure-boundary fields | active |
| `execute-from-spec` | Execute implementation work from an active task spec | A spec exists and work is ready to move into implementation | active spec, directly relevant repository context, validation path, and current control-surface context when used | smallest coherent in-scope change, explicit validation result, bounded repair or escalation outcome, and closeout decision | active |
| `design-whitebox-tests` | Decide whether and how to add white-box protection | A task touches complex internal logic or regression-sensitive code | task spec, bug context, internal logic details | white-box test decision, protected logic targets, regression guidance | active |
| `session-closeout` | Decide what durable updates, if any, should happen after a slice completes | A task or slice is finishing and control-surface, project-memory, fact, or skill write-back may be needed | completed task or slice, current control surface and project-memory state, any decision or experiment outcome, and candidate reusable context | targeted control-surface or project-memory updates, optional decision or experiment records, optional fact or skill updates, or an explicit no-write-back outcome | active |
| `replan-or-escalate` | Decide whether execution should continue, repair, roll back, replan, or stop for a decision | A task boundary, validation path, dependency, or policy choice becomes unsafe or unclear during execution | active spec, current failure or ambiguity, allowed/disallowed edits, repair budget, rollback scope, and escalation conditions | continue/repair/rollback/`replan_required`/`needs_decision` outcome with a short handoff reason | active |
| `decision-recording` | Create concise project decision records for durable choices | A choice changes project direction, scope boundary, control model, or durable operating rule | decision context, options, chosen option, reasoning, impact, and revisit trigger | decision record under `project/decisions/*` and optional `project/CURRENT.md` pointer | active |
| `autopilot` | Run a bounded autonomous execution loop without bypassing spec-first execution | The user asks the agent to continue through a clear reviewable slice with minimal interruption | active spec or trivially narrow request, repository rules, relevant files, validation path, repair budget, and escalation conditions | bounded change, validation result, repair/escalation outcome when needed, and closeout decision | active |
| `init-plastic-scm` | Bootstrap a copied project that uses Plastic SCM so AI tools stop defaulting to Git and stop opening the Plastic GUI | First entry into a copied project where `.plastic/plastic.selector` exists and the project's `AGENTS.md` does not yet declare Plastic as the backend | `.plastic/plastic.selector` marker, existing `.gitignore` / `.gitkeep` files, the three init templates, and the project's `AGENTS.md` / `project/CURRENT.md` | generated `ignore.conf`, generated `.cursorignore`, a project-specific `## Version control` section in the project's `AGENTS.md`, updated `project/CURRENT.md` VCS declaration, removed `.gitignore` and `.gitkeep`, and an init report with next-step tasks | active |
| `init-project-from-brief` | Turn a freshly copied SOP starter into a project-specific initial state using a project brief, so later work enters `plan-to-spec` against real project context | First entry into a copied starter when a brief is available or the user is willing to answer structured questions | one or more brief documents (or Q&A answers), the copied starter's `AGENTS.md` / `README.md` / `project/*` / `ai/doc/facts/project-scope.md`, relevant templates, and downstream skills (`init-plastic-scm`, `plan-to-spec`) | personalized `AGENTS.md` `## Repository purpose`, personalized `README.md` identity header, initialized `project/CURRENT.md` and `project/DOC_MAP.md`, seeded `ai/doc/facts/project-scope.md`, delegated Plastic bootstrap when applicable, and an init report with next-step handoff to `plan-to-spec` | active |
